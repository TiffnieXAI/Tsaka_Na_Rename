# Centry FastAPI Backend — Auth + User Schema + Platform OAuth + AI Features

`User` accounts with JWT auth, plus OAuth 2.0 connections to Shopee, Lazada,
and TikTok Shop (`PlatformConnection`, linked to a `User`).

Also includes **template/scaffold** implementations of the AI-Analysis-Layer
features from the flowchart: **Live Voice Verification** (deepfake/scam-call
detection), real-time **Cross-Platform Monitoring / behavioral trust scoring**
(delivered to the dashboard over a RabbitMQ-backed WebSocket), and the
**Centry AI Chatbot** (localized post-incident explanations). The API
contracts, DB models, and orchestration logic are fully built out and tested
end-to-end; the actual AI models are stubbed (see "AI feature templates"
below) for the AI team to wire in.

## Setup

```bash
pip install -r requirements.txt
cp .env.example .env
# edit .env with your real MySQL credentials + platform app credentials
```

## Database migrations (Alembic)

Schema is managed with Alembic against MySQL - `Base.metadata.create_all` is
**not** used against MySQL anymore (see `app/main.py`'s lifespan).

```bash
# apply all migrations (creates the schema from scratch on a fresh DB)
alembic upgrade head

# after changing app/models/models.py, generate the next migration
alembic revision --autogenerate -m "add whatever_you_changed"
# then read the generated file in alembic/versions/ before committing it -
# autogenerate is a good first draft, not a guarantee (enum changes, renames
# vs. drop+add, and data migrations all need a human look)

# roll back one revision
alembic downgrade -1
```

`alembic/env.py` builds the DB URL from the same `CENTRY_*` env vars / `.env`
the app itself uses (via `app.config.settings`) - there's nothing to
duplicate or keep in sync in `alembic.ini`.

The very first migration (`alembic/versions/20260714_0001_initial_schema.py`)
was hand-written to mirror `app/models/models.py` rather than autogenerated,
since it was created without a live MySQL instance to autogenerate against.
**Before relying on it against a real database**, point `CENTRY_DB_*` at an
empty MySQL instance, run `alembic upgrade head`, then run
`alembic revision --autogenerate -m check` once - if it generates an empty
migration, the hand-written one matches the models exactly; if not, fix
whichever one is wrong and delete the empty autogenerated file.


### Quick local test without MySQL

```bash
CENTRY_USE_SQLITE=true uvicorn app.main:app --reload
```

This path still uses `create_all` (see `app/main.py`) for a fast dev loop -

### Real-time events (RabbitMQ)

`/ws/events` needs a running RabbitMQ instance:

```bash
docker run -d --name centry-rabbitmq -p 5672:5672 -p 15672:15672 rabbitmq:3-management
```

The API starts fine without it (`CENTRY_RABBITMQ_OPTIONAL=true` by default) -
you'll just see a warning in the logs and `/ws/events` connections will sit
open with no events delivered until RabbitMQ is reachable.

## AI feature templates

### 1. Live Voice Verification (deepfake detection) — flowchart section 2

`app/routers/verification.py` exposes `WS /ws/live-verify`. Browser
WebSocket clients can't set an `Authorization` header, so auth is via a
`token` query param instead (`get_current_user_ws` in `app/core/deps.py`).

Protocol:
1. Client connects to `wss://host/ws/live-verify?token=<jwt>`.
2. Server accepts, creates a `VerificationSession` row, sends one
   `{"type": "session_initialized", ...}` frame.
3. Client sends raw binary audio chunks (confirm exact encoding/chunking
   with the frontend team - raw PCM16 vs WebM/Opus, chunk duration, etc.).
4. For each chunk, the server runs the two AI models **in parallel**
   (`asyncio.gather`) via `app/services/verification_service.py`, aggregates
   their output (`app/services/risk_engine_service.py`), persists a
   `RiskAssessment` row, and pushes back a `{"type": "risk_update", ...}`
   frame (see `LiveVerifyMergePayload` in `app/schemas/verification.py`)
   containing the combined risk score/level plus a `recommend_autonomous_block`
   flag for section 3 of the flowchart.
5. Client sends the text message `"END"` or disconnects to stop; the server
   marks the `VerificationSession` `COMPLETED`.

**Where the AI plugs in:** `app/services/ai_models/voice_biometrics.py` and
`app/services/ai_models/semantic_intent.py` are stubs returning
plausible-looking random scores. Swap the body of each `analyze()` for a
real model call — the function signatures and return dataclasses are the
contract the rest of the backend depends on.

### 2. Real-Time Dashboard Events (RabbitMQ) — flowchart sections 1 & 3

`app/routers/events.py` exposes `WS /ws/events` - the dashboard's live feed
of security happenings across every connected platform (Shopee, Lazada,
TikTok Shop, Meta): trust score updates, flagged incidents, autonomous
actions taken. It's a **separate** WebSocket from `/ws/live-verify` - that
one streams audio for a single call; this one streams general account
activity for as long as the dashboard tab is open.

**Why RabbitMQ, and not just a plain in-process WebSocket like
`/ws/live-verify`:** trust-score/incident events don't originate in the
same request that delivers them. A webhook can land on any uvicorn worker;
the AI team's behavioral model might run as its own service; a background
job might fire an alert at 3am with nobody making a live request at all.
None of those have a direct handle on which worker (if any) is holding a
given user's open dashboard socket. RabbitMQ decouples "something happened"
from "tell whoever's watching" - any process just publishes, and the
`/ws/events` connection (wherever it happens to be) picks it up.

Pipeline (flowchart steps 01-05, then 22-26 for section 3):
1. `POST /api/v1/webhooks/platform-event` (or, eventually, real
   per-platform webhook receivers) ingests raw platform activity telemetry.
2-3. `app/services/ai_models/behavioral_engine.py`'s `process_event()`
   scores it into a trust score / flagged reasons — **this is the AI
   team's plug-in point**, currently a stub, same pattern as the Live
   Verify models below.
4-5. `app/core/messaging/event_bus.py`'s `publish_event(user_id, event_type,
   data)` puts the result on a topic exchange (`centry.dashboard_events`,
   routing key `user.<id>.<event_type>`).
6. `app/routers/events.py` relays it to that user's open `/ws/events`
   connection(s), if any.

**Where the AI team plugs in:** `behavioral_engine.process_event()` -
returns random-but-plausible scores today so the whole webhook → RabbitMQ →
WebSocket → dashboard pipeline is testable without a real model. See the
file's docstring for the three ways to wire in the real implementation
(direct call, HTTP/gRPC to a separate service, or handing off to a task
queue if it's too slow to run inline). The same "keep the function
signature stable" contract applies as the other AI stubs.

**Testing the pipeline locally** (no real platform webhook or AI model
needed): start RabbitMQ, open a WebSocket to `wss://host/ws/events?token=<jwt>`
(e.g. via a small script, Postman, or `websocat`), then in another
terminal/tab call `POST /api/v1/webhooks/platform-event` with a body like:

```json
{
  "platform": "SHOPEE",
  "event_type": "login",
  "payload": {"ip": "203.0.113.5", "device": "unrecognized-android-14"}
}
```

You should see a `trust_score_updated` frame (and a `dashboard_sync` frame)
arrive on the `/ws/events` connection within milliseconds.

### 3. Centry AI Chatbot / chat logs — flowchart section 4

`app/routers/chat.py` exposes chat-session CRUD plus messaging.
`GET /api/v1/chat/sessions/{id}/logs` returns the full conversation with
metadata (`ChatLogResponse` in `app/schemas/chat.py`): each message's
language, which model answered, latency, and which `RiskAssessment` rows
were used as grounding context.

Sending a message (`POST /api/v1/chat/sessions/{id}/messages`) does:
1. Persists the user's message.
2. If the chat is linked to a `verification_session_id`, pulls that
   session's `RiskAssessment` history into a short context summary
   (`_build_incident_context` in `app/services/chat_service.py` —
   flowchart step 29, "query incident logs for context").
3. Calls the localized chatbot model with that context.
4. Persists + returns the assistant's reply.

**Where the AI plugs in:** `app/services/ai_models/localized_chatbot.py`'s
`generate_reply()` is a stub returning a canned, clearly-labeled placeholder
in the requested language (`en`/`fil`/`tgl`). Swap it for a real localized
LLM call.

### New DB models (`app/models/models.py`)

- `VerificationSession`, `RiskAssessment` — one row per Live Verify call,
  one row per analyzed audio window within that call.
- `ChatSession`, `ChatMessage` — one row per chatbot conversation, one row
  per turn, optionally linked back to a `VerificationSession`.

Covered by the initial Alembic migration (`alembic upgrade head`) - or by
`create_all` automatically if you're using `CENTRY_USE_SQLITE=true` for
local dev.

## How the OAuth flow works

1. Frontend calls `GET /api/v1/oauth/shopee/authorize` (with the user's Bearer token) and gets back `{"authorize_url": "..."}`.
2. Frontend redirects the browser to that URL. The user logs into Shopee/Lazada/TikTok Shop directly on *their* site and approves access.
3. That platform redirects the browser to our `/callback` endpoint with a `code` (and for Shopee, also a `shop_id`).
4. We exchange that code for an access token by calling the platform's token endpoint server-to-server, then save a `PlatformConnection` row.

**The tricky part: step 3's request has no Authorization header** — it's the *platform's* redirect, not an authenticated call from our own frontend. So how do we know which user this callback belongs to?

**Answer: the `state` parameter.** In step 1, we generate a short-lived signed JWT (`create_oauth_state_token`) encoding the user's ID and which platform they're connecting. We smuggle it into the redirect URL we send the platform (as `?state=...` on our own callback URL, since the platform echoes back whatever redirect URI we gave it). When the callback fires in step 3, we decode that `state` token to recover the user ID — no session, no cookie, no database lookup needed to know who's connecting.

This is verified working end-to-end (tested with the token-exchange step mocked, since this sandbox can't reach Shopee/Lazada/TikTok's real servers) — see the flow described above for the exact mechanics.

on exactly what's confirmed vs. not.

## ⚠️ Before deploying anywhere real

- **Change `jwt_secret_key`** — generate with `openssl rand -hex 32`.
- **Platform tokens are stored in plain text** in `PlatformConnection` — encrypt before this touches real seller accounts.
- **No token refresh logic yet** — Shopee tokens expire in ~4 hours, Lazada/TikTok Shop have their own expiries. `token_expires_at` is stored but nothing currently uses it to proactively refresh.
- **No password reset / email verification flow yet.**
- **The `/callback` endpoint has no rate limiting or replay protection** beyond the state token's expiry - fine for now, worth hardening later.
- **`alembic/versions/20260714_0001_initial_schema.py` was hand-written, not autogenerated** - verify it against a real MySQL instance before depending on it (see "Database migrations" above).
- **`POST /api/v1/webhooks/platform-event` is a stand-in**, not a real per-platform webhook receiver - no signature verification, and it trusts the caller's own JWT to identify the user rather than resolving `external_account_id` -> user via `PlatformConnection`. Replace before connecting real platform webhooks (see the endpoint's docstring).
- **No running trust score is persisted yet** - each `/api/v1/webhooks/platform-event` call scores from a blank baseline (`previous_trust_score=None`). Add a model for it once the AI team's real behavioral engine needs a real trend to compare against.
- **RabbitMQ events are `NOT_PERSISTENT`** (`app/core/messaging/event_bus.py`) - fine for "live UI push," but if events ever need to survive a RabbitMQ restart or be replayed, switch to persistent delivery and a durable queue per subscriber.

## Project structure

```

app/
  main.py                    - FastAPI app setup, lifespan (RabbitMQ connect/close, dev-only create_all), router registration, exception handlers
  config.py                  - all settings (DB, JWT, OAuth per-platform, RabbitMQ), loaded from .env

  models/models.py            - User, PlatformConnection, VerificationSession, RiskAssessment, ChatSession, ChatMessage
  schemas/
    auth.py                  - register/login/me shapes

    verification.py            - Live Verify REST + WebSocket payload shapes
    chat.py                     - chat session/message/log shapes
    events.py                    - EventType enum + PageEvent envelope for /ws/events
    webhooks.py                   - PlatformActivityIn (platform-event ingestion request shape)
  core/
    security.py                - password hashing, JWT create/decode, OAuth state token create/decode
    deps.py                     - get_current_user / get_current_user_ws dependencies
    exceptions.py                - DuplicateResourceException (409), ResourceNotFoundException (404)
    oauth/
      base.py                   - OAuthProvider abstract class + TokenResult shape
      shopee.py                  - Shopee implementation (verified)
      lazada.py                   - Lazada implementation (verified)
      tiktok_shop.py               - TikTok Shop implementation (⚠️ unverified)

      registry.py                    - platform -> provider factory
    messaging/
      rabbitmq.py                 - shared connection/exchange lifecycle (connect_rabbitmq/close_rabbitmq)
      event_bus.py                  - publish_event() / subscribe_user_events() on top of the topic exchange
  services/
    auth_service.py               - register/authenticate logic
    oauth_service.py                - start_authorization, complete_authorization, connection CRUD
    verification_service.py          - Live Verify session lifecycle + per-chunk analysis orchestration
    risk_engine_service.py            - aggregates voice + semantic model output into one risk score/level
    chat_service.py                    - chat session/message orchestration, incident-context building
    ai_models/                          - STUB / TEMPLATE layer - AI team's real code plugs in here

  routers/
    auth.py                        - the 3 auth endpoints
    oauth.py                        - the 4 oauth endpoints
    verification.py                  - WS /ws/live-verify + Live Verify session history endpoints
    chat.py                           - chatbot session/message/log endpoints
    events.py                          - WS /ws/events (real-time dashboard events)
    webhooks.py                         - POST /api/v1/webhooks/platform-event (platform telemetry ingestion)
```
